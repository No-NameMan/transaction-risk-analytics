# Transaction Risk Analytics

A compact analytics project that simulates banking transactions and identifies suspicious users and merchants using SQL, pandas, exploratory data analysis, peer-group comparison, and a simple heuristic risk score.

The project is designed as a GitHub-ready MVP for an analyst internship portfolio. It focuses on explainable business logic rather than complex machine learning.

## Project idea

The goal is to analyze transaction behavior in a banking-like context and identify objects that may require additional review.

The analysis focuses on two types of objects:

* merchants with unusually high turnover, average ticket, transaction count, or night activity compared with similar merchants;
* users with unusually high transaction volume, burst-like behavior, night activity, or broad merchant coverage compared with similar users.

The key analytical idea is peer-group comparison:

* merchants are compared within `category + city`, with a fallback to broader `category` peer groups when the narrow peer group is too small;
* users are compared within `segment + city`.

## Data

The dataset is synthetic and generated by `src/generate_data.py`.

This is intentional: the project does not use real banking data, personal data, or confidential transaction records.

Generated tables:

* `users.csv` — synthetic bank customers;
* `merchants.csv` — synthetic merchants;
* `transactions.csv` — synthetic transactions;
* `injected_cases.csv` — synthetic injected risk scenarios used only for sanity checks.

The risk score does not use the injected labels directly. It is based on observable transaction behavior.

## Methodology

### 1. Data generation

The project generates synthetic users, merchants, and transactions.

The generator also injects several artificial risk scenarios:

* merchants with unusually high turnover and night activity;
* users with burst-like transaction activity over a short period.

These scenarios make it possible to check whether the analytical logic can detect suspicious behavior.

### 2. SQL analysis

SQL is used to calculate aggregated risk metrics.

Merchant-level metrics include:

* transaction count;
* approved turnover;
* average ticket;
* number of unique users;
* night transaction share;
* declined transaction share;
* turnover relative to category-city peer group;
* average ticket relative to category-city peer group;
* transaction count relative to category-city peer group.

User-level metrics include:

* transaction count;
* total approved amount;
* average ticket;
* number of distinct merchants;
* active days;
* maximum daily transaction count;
* night transaction share;
* declined transaction share;
* total amount relative to segment-city peer group;
* transaction count relative to segment-city peer group.

### 3. Risk scoring

The risk score is a simple explainable heuristic.

For merchants, the score is based on:

* high turnover compared with peers;
* high average ticket compared with peers;
* high transaction count compared with peers;
* high night transaction share;
* high declined transaction share.

For users, the score is based on:

* high total amount compared with peers;
* high transaction count compared with peers;
* burst-like daily activity;
* high night transaction share;
* transactions across many merchants;
* high declined transaction share.

The score is not a machine learning model. It is a transparent rule-based indicator designed for analytical prioritization.

### 4. Peer-group comparison details

Merchant peer comparison uses a leave-one-out approach: the merchant is excluded from its own peer average. This prevents large outliers from inflating the benchmark used to evaluate themselves.

The default merchant peer group is `category + city`. If this group is too small, the analysis falls back to a broader `category` peer group. The output table includes `peer_group_level` and `peer_group_size` to make this logic transparent.

Synthetic injected cases are not used in scoring. They are used only after scoring as a sanity check to verify whether the heuristic risk score surfaces the intended suspicious scenarios in the top-N review queue.

## Example business interpretation

A merchant may require additional review if it has:

* turnover several times higher than similar merchants in the same category and city;
* unusually high average ticket;
* a high share of night transactions;
* a high number of unique users over a short period;
* an increased declined transaction share.

A user may require additional review if they have:

* transaction volume much higher than similar users;
* burst-like activity over a small number of days;
* many transactions at night;
* transactions across many unrelated merchants;
* a high declined transaction share.

These signals do not prove fraud. They indicate that the object is atypical and should be prioritized for further investigation.

## What additional data would be useful in a real banking task

In a real risk analytics setting, the analysis should be enriched with:

* chargebacks;
* refunds;
* disputes;
* customer complaints;
* confirmed fraud labels;
* KYC data;
* device fingerprints;
* geolocation;
* merchant onboarding information;
* transaction limits;
* historical account behavior;
* external blacklists or watchlists.

## Limitations

* The data is synthetic.
* The score is heuristic and not calibrated on real fraud labels.
* Peer groups are simplified.
* The analysis does not estimate financial loss.
* The project does not include production monitoring, model deployment, or real-time decisioning.
